# MRO-015: Top-level decomposition pipeline (stack-decompose)

## Description
Implement the Level 0 pipelines that decompose a MatrixRTCStack into its constituent intermediate CRDs: LiveKitStack + MatrixStack. Plus reverse status aggregation from child CRDs back to the parent MatrixRTCStack.

This is the novel architectural contribution of matrix-rtc-operator — cascading single panes of glass where a single top-level resource drives the entire stack, and status propagates back up.

## Acceptance Criteria
- [ ] `stack-decompose.yaml` creates both LiveKitStack and MatrixStack CRs from a MatrixRTCStack
- [ ] `stack-status.yaml` aggregates status from child CRDs back to MatrixRTCStack
- [ ] infraRef auto-discovery: if MatrixRTCStack.spec.infraRef is not specified, automatically find the single RTCInfrastructure in the cluster
- [ ] Apply a MatrixRTCStack CR -> both LiveKitStack and MatrixStack CRs appear in the same namespace
- [ ] Deleting the MatrixRTCStack CR cascades deletion to child CRs
- [ ] Child CRs have ownerReferences pointing to parent MatrixRTCStack

## Notes
- This is a pure dcontroller pipeline — no Go code, just declarative YAML.
- The decomposition pattern: MatrixRTCStack spec is split into relevant sections for each child CR. LiveKitStack gets livekit.* config, MatrixStack gets matrix.* config, both get infraRef.
- Status aggregation should show: number of ready components, overall phase (Pending/Provisioning/Ready/Degraded), and conditions from child CRs.
- infraRef auto-discovery uses a dcontroller query: list RTCInfrastructure resources, if exactly 1 exists, use it. If 0 or >1, set error condition on MatrixRTCStack.
