# MRO-012: CRD templates (MatrixRTCStack, LiveKitStack, MatrixStack, RTCInfrastructure)

## Description
Define all 4 Custom Resource Definitions as Helm templates (placed in templates/, not the crds/ directory). This allows lifecycle management via Helm hooks and annotations. Include OpenAPI v3 validation schemas for each CRD.

- **MatrixRTCStack** (namespaced) — user-facing top-level resource combining Matrix + LiveKit + infrastructure
- **LiveKitStack** (namespaced) — intermediate CRD for LiveKit components
- **MatrixStack** (namespaced) — intermediate CRD for Matrix components
- **RTCInfrastructure** (cluster-scoped) — shared infrastructure config (gateways, cert-manager, STUNner)

## Acceptance Criteria
- [ ] All 4 CRDs defined as Helm templates under templates/crds/
- [ ] `crds.create` toggle in values.yaml controls whether CRDs are rendered
- [ ] `helm.sh/resource-policy: keep` annotation on all CRDs (prevent accidental deletion)
- [ ] ArgoCD sync-wave annotation (CRDs install before operator)
- [ ] OpenAPI v3 validation for required fields on each CRD
- [ ] MatrixRTCStack, LiveKitStack, MatrixStack are namespaced; RTCInfrastructure is cluster-scoped
- [ ] CRD group: lukacsi.org, version: v1alpha1

## Notes
- CRDs in templates/ (not crds/) because: crds/ doesn't support templating, can't be toggled, and has no upgrade lifecycle.
- Views (LiveKitServerView, MatrixServerView, etc.) do NOT need formal CRDs — dcontroller creates them dynamically at runtime.
- Sync-wave: CRDs should be wave -1 or lower so they exist before the operator Deployment tries to watch them.
